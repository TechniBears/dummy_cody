Memory Reality

What is actually true right now, not what the most ambitious README implies.

Working

claude-flow local memory is real: semantic search works, the Agent Cody namespace is present, and the repo is carrying project state in a local sql.js + HNSW store.

Not Fully Live

The deeper AgentDB bridge is not available in this session, and the embeddings subsystem is not initialized. The architecture exists, but not all of it is actually operating.

Finicky Behavior

Observed locally: stale HNSW metadata can keep older versions of a key around even when SQLite stays clean. Result: duplicate semantic hits. The store works, but the vector index needs hygiene.

Cody-Specific Truth

Current Cody runtime memory is still OpenClaw builtin. The right business-memory target remains Graphiti, because Cody cares about changing facts over time, not just nearest-neighbor recall.

Use ./scripts/cody-studio.sh memory-doctor to inspect the local claude-flow store, and ./scripts/cody-studio.sh graphiti smoke to test the Neo4j-backed graph path.

Compare Memory Options

Pick the operating lens. The recommendation stays stable, but the tradeoffs change depending on whether you care about shipping now, operator ergonomics, or research ambition.

Best fit now

Graphiti / Zep

Best primary memory for Cody because it models entities, relationships, and validity over time. That is the hard problem in a sales-and-email operator.

  • Strongest fit for deal state, contact state, and temporal corrections
  • Supports a real memory explorer and timeline later
  • Operationally heavier than a pure vector store, but the model is right
Useful support layer

claude-flow / AgentDB / RuVector

Strong as a secondary engineering and operator memory. Fast local search. Good place for deployment lessons and recovery patterns. Not the truth store for live customer facts yet.

  • Useful for project memory and pattern recall
  • Current local install shows partial availability and index quirks
  • Keep it beside Cody, not underneath Cody's business memory
Too generic

OpenClaw builtin only

Fine for short-lived continuity. Wrong as the long-term memory of a business operator because it does not natively express fact invalidation and time-aware truth.

  • Keep as session cache
  • Do not trust as source of truth
  • Good enough while Graphiti is not wired yet
Best operator surface

Letta

If you wanted Cody to feel more like an editable agent OS with explicit memory blocks and rich state control, Letta is attractive. But it would be a runtime commitment, not a small add-on.

  • Great for visible, editable agent state
  • Heavier shift away from the OpenClaw-centered plan
  • Better for agent state than temporal deal truth
Best business fit

Graphiti / Zep

Still the right primary memory. The fact that it is slightly heavier operationally is worth it because it matches the actual business semantics.

  • Best for timeline-aware briefing and drafting
  • Best for contact and deal graphs
  • Most compatible with the dashboard memory explorer you already designed
Practical helper

Mem0

Fast way to add memory behavior with less operational weight than a graph system, but it is easier to end up with "best matching memory" where you really wanted "currently true fact".

  • Lower operational friction
  • Less aligned with temporal business state
  • Good if you need simpler memory before Graphiti is ready
Interesting future accelerator

RuVector / AgentDB / GNN stack

The most interesting research layer here. If you later want learned routing, faster vector analytics, or structural pattern retrieval over operator memory, this is worth revisiting.

  • Good for pattern memory and developer workflows
  • Promising for analytics and retrieval acceleration
  • Still too ambitious to be Cody's live customer-memory foundation today
Composable library path

LangMem

If Cody later drifts toward a more custom LangGraph-based orchestration layer, LangMem becomes interesting as a lighter memory primitive.

  • Good if the orchestration layer changes
  • Not the shortest path from today's OpenClaw setup
  • Lower commitment than a full agent runtime swap
Still the business winner

Graphiti / Zep

Even in research mode, Graphiti remains the better primary memory for facts that change over time. GNN retrieval does not replace temporal validity.

  • Primary truth store stays graph-temporal
  • Experimental vector systems can sit beside it
  • Keep the source of truth boring and correct

Recommended Stack

The memory split I would actually run.

Layer 1 - OpenClaw builtinShort-lived session memory only. Let it keep immediate continuity. Do not make it authoritative.
Layer 2 - Graphiti / Neo4jPrimary business memory. Deals, contacts, preferences, historical corrections, and timeline-aware facts live here.
Layer 3 - claude-flow memorySecondary engineering and operator memory. Keep platform lessons, AWS incidents, debugging patterns, and repo decisions here.
Layer 4 - RuVector later, if earnedUse only after you prove a real need for faster search or pattern analytics. It is an accelerator layer, not the first layer to trust.

Growth Path

How to grow Cody without turning the system into a mess.

Now

Stabilize text replies, edit prompt files directly, and stop burying system behavior in bootstrap scripts.

  • Use workspace/SOUL.md and workspace/AGENTS.md as source of truth
  • Keep memory simple while Bedrock and Telegram stabilize
  • Use claude-flow memory for project lessons, not customer truth

Next

Make Graphiti real and visible.

  • Use memory-read and memory-write against the graphiti-memory sidecar
  • Add a memory explorer in the dashboard
  • Add a diff-based config commit flow for prompt changes

Later

Add acceleration only after the truth model is correct.

  • Benchmark RuVector against real Cody workloads
  • Consider pattern memory over operator incidents and draft revisions
  • Keep temporal business truth separate from experimental retrieval layers
